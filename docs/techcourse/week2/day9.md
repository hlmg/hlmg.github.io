# 프리코스 9일 차 10/27

코드 리뷰를 다 하지 못해서 나머지 리뷰를 남기고 내가 받은 코드 리뷰를 토대로 숫자 야구 게임 리팩토링을 진행했다.

## 숫자 야구 리팩토링

리팩토링할 내용은 가독성 개선, 불필요한 형변환 개선, 기타 코드 스타일이 있었다.

가독성 개선 내용은 볼, 스트라이크 개수를 셀 때 스트림을 두 번 돌렸는데, 가독성이 좋지 않다는 코멘트가 있었다. 공감되는
내용이라 내부 스트림을 메소드로 분리해서 가독성을 개선했다.

불필요한 형변환 개선 내용은 View와 Converter사이에서 형변환이 `String -> int -> String` 이렇게 되고있었는데, 불필요하지 않냐는
코멘트였다. 처음에 이 내용을 보고 '왜 이렇게 작성했지?'라는 생각이 들었고, View에서 숫자 검증을 한 뒤에 Converter를 직접 사용해서 `List<Integer>`를 반환하게 변경했다.

## 자동차 경주 게임

자동차 경주 게임 미션 메일에 공통 피드백과 숫자 야구 강의가 있었다. 공통 피드백이어서 그런지 대부분이 컨벤션에 관련된 내용이었다. 숫자 야구 강의까지 듣고 자동차 경주 게임 미션을 진행했다.

이번 미션엔 추가 요구사항이 있었다.

- indent(인덴트, 들여쓰기) depth를 3이 넘지 않도록 구현한다. 2까지만 허용한다.
- 3항 연산자를 쓰지 않는다.
- 함수(또는 메서드)가 한 가지 일만 하도록 최대한 작게 만들어라.
- JUnit 5와 AssertJ를 이용하여 본인이 정리한 기능 목록이 정상 동작함을 테스트 코드로 확인한다.

다음 미션들에도 계속해서 요구사항이 추가될 것 같다는 느낌이 들었다. 아직 난이도가 높은 요구사항은 없는 것 같아서, 꼼꼼하게만 진행하면 될 것 같다.

구현 전에 README에 구현할 기능 목록을 작성했다. 이번에는 저번 주 미션에서 했던 실수를 만회하기 위해 시스템에 전송되는 메세지를 생각하면서 기능 목록을 작성했다.

:::details 기능 목록

- [ ] 자동차 게임 시작하기
- [ ] 자동차 경주 게임에 참여할 참가자 생성하기
    - [ ] 경주할 자동차 이름 입력받기
        - 이름은 쉼표(,) 기준으로 구분한다
        - 5자 이하만 가능하다
    - [ ] 참가자 생성하기
- [ ] 자동차 경주 진행하기
    - [ ] 시도할 횟수 입력 받기
        - 입력 값이 숫자가 아니면 에러 발생
        - 숫자는 1 이상이어야 한다
    - [ ] 자동차 움직이기
        - 0에서 9 사이의 무작위 값이 4 이상이면 이동한다
    - [ ] 차수별 실행 결과 출력하기
- [ ] 우승자 안내하기
    - [ ] 우승자 목록 가져오기
    - [ ] 우승자 출력하기
- 우승자가 여러 명이면 쉼표(,)를 이용하여 구분한다

:::
